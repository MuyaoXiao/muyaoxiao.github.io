我对IO的恐慌由来已久，我学过Java的IO，它告诉我有字节流啦字符流啦然后分为io package以及后来的nio package。What the fuck? 这些到底都是什么？有时候我只是想读一个文件而已，但它给了我太多接口，我不知道要用哪一个。

Go的io 包的核心就是围绕两个interface建立的，也就是Reader和Writer。至少形式上要比Java简化很多：

但是这依旧让人迷糊，因为对于一个不熟悉操作系统的人来说，请问Reader是从哪儿read到哪儿？Writer又是从哪儿write到哪儿？为什么读一个文件需要这两个接口？

这类问题的来源是对Unix所知甚少，至少是认识不够深刻。Unix的哲学是:

Make each program do one thing well. (Doug McIlroy)

因此「读取」，「打印（写在console上）」这两件事情自然需要有两个东西去完成，Write和Read的分野顺理成章。

Unix的另一个哲学是：

Expect the output of every program to become the input to another, as yet unknown, program. (Doug McIlroy)

这两句话可以说贯穿了整个io包的设计思路，毕竟Ken Thmopson和Rob Pike，你懂的。

所以，go的文件读写流程可以参考此文：

1

File -> io.Reader -> []byte -> io.Writer -> Target

reader.Read(b []byte)的意思是将什么东西读入b []byte当中，而writer.Write(b []byte)则是将b []byte写到什么上。

那么，读取文件，自然在os.Open(path)之后要返回一个reader，事实上标准包中返回了一个File, 它存在Read方法。

再比如，输出到标准输出的fmt.Println()必然也应当是调用一个Writer将[]byte写到stdout中，事实上:

func Println(a ...interface{}) (n int, err error) {
	return Fprintln(os.Stdout, a...)
}

func Fprintln(w io.Writer, a ...interface{}) (n int, err error) {
	p := newPrinter()
	p.doPrintln(a)
	n, err = w.Write(p.buf)
	p.free()
	return
}

简单来说，记忆io应当遵循Unix的哲学。当然Unix哲学并非唯一的哲学，但至少在go这里我觉得是基本适用的，尽管我认为它应当更直观点。
